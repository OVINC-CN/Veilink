# Veilink

[![CI](https://github.com/OVINC-CN/Veilink/actions/workflows/ci.yml/badge.svg)](https://github.com/OVINC-CN/Veilink/actions/workflows/ci.yml)
[![Container images](https://github.com/OVINC-CN/Veilink/actions/workflows/images.yml/badge.svg)](https://github.com/OVINC-CN/Veilink/actions/workflows/images.yml)

Privacy-first, ephemeral browser chat with end-to-end encrypted messages and
files. Veilink has no database and does not upload chat content to the
application server. A room exists only while at least one authenticated browser
tab remains connected.

[简体中文](README_CN.md)

> **Security status:** this project is an early implementation. Review the
> threat model and deployment settings before using it for sensitive work.

## What it does

- Joins rooms through a link plus a separately shared six-digit PIN.
- Derives E2EE keys locally from the link secret and PIN; the URL fragment is
  removed from the address bar immediately after it is read.
- Sends chat data over TURN-relayed WebRTC DataChannels. Fastify only coordinates
  room membership, WebRTC signalling, and short-lived relay credentials.
- Forces every WebRTC connection through TURN so participants never establish
  direct connections or expose their public IPs to one another.
- Keeps rooms and signalling state only in process memory. Vacant rooms remain
  available until expiry; owner destruction or an application restart also
  destroys the room.
- Renders constrained structured rich text, privacy-preserving local link cards,
  and in-memory previews for supported attachments.

## Architecture

The pnpm workspace contains a React/Vite browser client, a Fastify signalling
server, and a shared protocol package. The production container serves the
compiled client and API from one origin. Coturn uses short-lived REST/HMAC
credentials generated by the application from a shared deployment secret.

There is intentionally no database, cache service, object store, upload route,
server-side message history, or writable data volume. The deployment supports
exactly one application replica. Do not run `docker compose up --scale app=...`; replicas
cannot share room state. Restarting the app invalidates every active room.

## Requirements

- Node.js 22 and pnpm 11 for local builds
- Docker Engine with Docker Compose v2 for container deployment
- A public IPv4 address with inbound UDP/TCP 3478 and UDP relay ports
  49160–49200 forwarded to the TURN host
- DNS names and a valid TLS certificate for the public chat origin

## Local verification

```bash
pnpm install --frozen-lockfile
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

The server runs from `apps/server/dist/index.js` after the build. HTTPS is
required outside localhost because WebRTC and cryptographic browser APIs require
a secure context.

## Container deployment

1. Copy and edit the environment template:

   ```bash
   cp .env.example .env
   openssl rand -hex 32
   ```

   Put the generated value in `TURN_REST_SECRET`. Set `APP_ORIGIN` to the exact
   public HTTPS origin, replace every example TURN hostname/IP, and keep the
   `.env` file outside source control with restrictive permissions.

2. Confirm the fixed Compose subnet does not overlap another host network. If
   it does, change `VEILINK_SUBNET`, `VEILINK_GATEWAY`, and the matching trusted
   proxy address together.

3. Validate and start the deployment:

   ```bash
   docker compose config --quiet
   docker compose build
   docker compose up -d
   ```

The application is published on `127.0.0.1:3000` by default and must sit behind
an external HTTPS reverse proxy. Coturn is public on UDP/TCP 3478 plus the
configured UDP relay range. No persistent Docker volumes are created; both
containers use read-only root filesystems and small tmpfs mounts for unavoidable
runtime files. The Compose file also disables Docker's container log driver so
stdout/stderr cannot become a second persistent connection log. Equal memory and
memory-plus-swap limits prevent container memory from being swapped when the
host's container runtime supports that control, and core dumps are disabled.

The app can alternatively terminate HTTPS itself. Set `TLS_CERT_FILE` and
`TLS_KEY_FILE` together to readable PEM paths inside the app container, mount
only those two files read-only, publish the app port on the intended interface,
and set `APP_ORIGIN` to that exact HTTPS origin. Leaving both variables empty
keeps the default reverse-proxy setup.

Compose builds a thin coturn image from the pinned `coturn/coturn:4.14.0-r0`
base. It copies the upstream binary without its file-capability xattr so the
container can retain `no-new-privileges` and `cap_drop: ALL`; otherwise Linux
rejects startup with exit code 126. No runtime capability is added.
Main-branch builds publish multi-platform images as
`ghcr.io/ovinc-cn/veilink` and `ghcr.io/ovinc-cn/veilink-turn`, with `latest`
and immutable `sha-*` tags plus SBOM/provenance attestations. A Git tag push
also publishes both images with the Git tag as the image tag. Production should
set `VEILINK_IMAGE` and `VEILINK_TURN_IMAGE` to immutable `sha-*` tags.

### Reverse proxy and trusted IPs

The proxy must support WebSocket upgrade on `/signal`, preserve the original
`Host`, and set `X-Forwarded-Proto: https`. Publish only HTTPS to users. A minimal
Nginx location resembles:

```nginx
location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto https;
    proxy_set_header X-Forwarded-For $remote_addr;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}
```

`TRUST_PROXY_CIDRS` must contain only the direct proxy hop. With the default
bridge and a proxy on the Docker host, that hop is `172.30.0.1/32`. Do not use
`0.0.0.0/0`, a broad private range, or blindly trust incoming forwarded headers.
If the proxy is another container, give it a stable address and trust only that
address.

**Disable access logs or irreversibly anonymize source IPs and URLs at the
reverse proxy.** Veilink cannot enforce the logging policy of infrastructure in
front of it. Do not enable Fastify, coturn, or Docker connection logs in
production. If diagnostics require temporary logging, use a short, documented
window and securely erase the resulting logs afterward.

### TURN networking

`TURN_EXTERNAL_IP` is the public address advertised to browsers. The entrypoint
automatically pairs it with the coturn container's private address; set
`TURN_PRIVATE_IP` only when automatic detection is wrong. The public firewall
and any upstream NAT must forward both 3478 transports and the exact UDP relay
range in `.env`.

TURN-over-TLS on TCP 5349 is optional. It terminates in coturn, not in an HTTP
reverse proxy. Mount a readable certificate and private key using the supplied
overlay:

```bash
docker compose -f docker-compose.yml \
  -f docker-compose.turns.yml.example config --quiet
docker compose -f docker-compose.yml \
  -f docker-compose.turns.yml.example up -d
```

Update `TURN_URLS` with a `turns:turn.example.com:5349?transport=tcp` entry when
enabling it. Certificate/key mounts are read-only and are the only optional host
mounts; they contain configuration secrets, not Veilink user data. Grant the
private key specifically to container UID 65534 with host ownership or an ACL;
do not make it world-readable.

## Configuration

| Variable | Purpose | Default / constraint |
| --- | --- | --- |
| `APP_ORIGIN` | Exact public browser origin | Required HTTPS URL |
| `VEILINK_IMAGE`, `VEILINK_TURN_IMAGE` | App and hardened coturn image tags | GHCR `latest`; pin `sha-*` in production |
| `TLS_CERT_FILE`, `TLS_KEY_FILE` | Optional native app HTTPS PEM paths | Set both or neither |
| `TRUST_PROXY_CIDRS` | Direct reverse-proxy CIDR(s) | Required, explicit |
| `ROOM_TTL_SECONDS` | In-memory room lifetime | `86400`, cannot exceed it |
| `TURN_URLS` | Comma-separated browser TURN URLs | Required |
| `TURN_REALM` | Shared TURN realm | `veilink` |
| `TURN_REST_SECRET` | App/coturn HMAC secret | Required; use 32 random bytes |
| `TURN_CREDENTIAL_TTL_SECONDS` | Temporary TURN credential lifetime (hard maximum: 600 seconds) | `600` |
| `MAX_CONNECTIONS` | Global WebSocket connection cap | `2048` |
| `MAX_CONNECTIONS_PER_IP` | WebSocket cap per source IP | `64` |
| `MAX_ROOMS_PER_IP` | Active rooms created per source IP | `32` |
| `ROOM_CREATE_ATTEMPTS_PER_MINUTE` | Room-create attempts per source IP/minute | `20` |
| `TURN_EXTERNAL_IP` | Public TURN IPv4 address | Required |
| `TURN_MIN_PORT`, `TURN_MAX_PORT` | Published UDP relay range | `49160`–`49200` |

The six-digit room PIN is not an environment secret and is never configured on
the server. It is generated per room and mixed with the link fragment entirely
in the browser.

## Privacy boundaries

Veilink avoids application-level persistence, but a normal browser and operating
system cannot guarantee that RAM, Blob implementations, swap, crash dumps,
browser history synchronization, screenshots, extensions, or downloaded files
never reach storage. Treat “no trace” as a best-effort application property, not
an operating-system guarantee.

The same boundary applies to the server host: Compose disables container swap
and core dumps where supported, but the operator remains responsible for host
hibernation, hypervisor snapshots, crash capture, and physical-memory policy.

The only intended browser persistence exceptions are:

1. Files a user explicitly chooses to download.
2. The `veilink.preferences.v1` localStorage record, containing only UI language,
   theme, file-size limit, send shortcut, timestamp display,
   density, and the optional remembered nickname preference/value.

Clicking a copy action also explicitly hands the PIN or invitation link to the
operating-system clipboard. Clipboard history and cloud synchronization are
outside browser control and may retain it after the tab closes; overwrite the
clipboard with non-sensitive content after sharing.

Room IDs, link secrets, PINs, derived keys, fingerprints, messages, links,
attachments, participant addresses, and reconnect tokens must never be placed in
localStorage, sessionStorage, IndexedDB, Cache API, or a service worker cache.
Closing or refreshing the tab clears application memory and revokes Blob URLs.

E2EE protects message and file content from the signalling and TURN servers. It
does not hide connection metadata from the infrastructure operator, protect a
compromised endpoint or browser extension, or protect users if the served
JavaScript itself is maliciously replaced. Forced relay prevents participants
from learning one another's public IP, but infrastructure operators can still
observe connection metadata.

TURN REST credentials and existing allocations cannot be revoked room by room.
The default credential and allocation lifetime is ten minutes. Destroying a
room removes application state immediately, while an existing TURN allocation
may survive until coturn times it out; it still cannot read E2EE content.

## Operational checklist

- Keep the app at one replica and expect every restart to destroy active rooms.
- Rotate `TURN_REST_SECRET` as a coordinated app/coturn deployment; rotation
  invalidates outstanding TURN credentials.
- Restrict `.env` and TLS-key permissions and never print their contents in CI.
- Disable/anonymize proxy, load-balancer, firewall, and DNS query logs where the
  privacy policy requires it.
- Patch Node.js, the base image, coturn, and dependencies regularly; run
  `pnpm audit --prod` and rebuild images after security updates.
- Back up no Veilink application data: there is intentionally nothing to restore.
